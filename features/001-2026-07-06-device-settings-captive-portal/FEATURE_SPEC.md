# Feature Spec: Device Settings Captive Portal

## Цель

Добавить на KAST локальную веб-страницу настроек, доступную через Wi-Fi точку доступа устройства, и обеспечить продолжение текущей row-сессии после выключения и повторного включения питания.

## Контекст и проблема

Сейчас настройки устройства доступны только через прошивку и поведение физических кнопок. Пользователю нужен простой способ менять параметры без перепрошивки: звук при включении, яркость экрана и режим сна.

Также текущая row-сессия не должна сбрасываться при выключении питания. Для вязания потеря текущего количества rows является критичной ошибкой, поэтому состояние должно переживать перезапуск устройства.

## Целевые пользователи

- Вязальщик, который использует устройство во время проекта и хочет быстро менять базовые настройки.
- Разработчик или тестировщик устройства, которому нужен ручной доступ к настройкам без сборки новой прошивки.

## Пользовательские сценарии

- Пользователь находится в основном меню, удерживает кнопку `+` 3 секунды, устройство включает Wi-Fi точку доступа с веб-страницей настроек.
- После включения Wi-Fi точки доступа экран переходит в цветовое состояние как на паузе, сверху показывает `Настройки`, а также отображает IP-адрес для ручного открытия страницы.
- Пользователь подключается к Wi-Fi устройства, ОС пытается открыть captive portal автоматически; если это не произошло, пользователь открывает ручной адрес `192.168.4.1`.
- Пользователь меняет настройки на веб-странице и нажимает `Save`; устройство применяет и сохраняет настройки.
- Пользователь повторно удерживает кнопку `+` 3 секунды в основном меню, устройство выключает Wi-Fi точку доступа.
- Пользователь меняет rows кнопками `+` или `-`, выключает устройство, затем включает его снова; row-сессия продолжается с последнего сохраненного значения.

## Функциональные требования

- Устройство должно уметь включать и выключать режим Wi-Fi точки доступа из основного меню долгим нажатием кнопки `+` на 3 секунды.
- Долгое нажатие `+` должно работать как переключатель: если точка доступа выключена, она включается; если включена, она выключается.
- Веб-страница настроек должна быть доступна через captive portal после подключения к Wi-Fi устройства.
- Если captive portal не открылся автоматически, страница должна быть доступна по ручному адресу `192.168.4.1`.
- После включения AP экран должен использовать такой же цвет фона или цветовую схему, как экран паузы.
- После включения AP экран должен показывать заголовок `Настройки` сверху.
- После включения AP экран должен показывать IP-адрес страницы настроек, чтобы пользователь мог открыть ее вручную, если captive portal не сработал.
- Страница настроек должна содержать настройку включения или выключения boot beep.
- Boot beep означает только звуковой сигнал при включении устройства, а не все возможные звуки устройства.
- Страница настроек должна содержать настройку яркости экрана в диапазоне от `0` до `100`.
- Значение яркости по умолчанию должно быть `50`.
- Страница настроек должна содержать настройку включения или выключения режима сна.
- Режим сна означает затухание экрана для экономии батареи, а не выключение устройства или завершение row-сессии.
- Страница настроек должна содержать время до затухания экрана для режима сна.
- Если режим сна выключен, экран не должен автоматически затухать по таймауту.
- Настройки должны применяться и сохраняться только после нажатия `Save`.
- Сохраненные настройки должны переживать перезагрузку и выключение питания.
- Текущая row-сессия должна сохраняться после каждого изменения rows.
- После выключения и повторного включения питания устройство должно восстановить текущую сессию с последним сохраненным row count.
- Существующая терминология UI и логики должна использовать `rows`.

## Нефункциональные требования

- Веб-страница должна быть простой и достаточно легкой для ESP32-S3.
- Настройки должны быть доступны без интернета и без внешнего роутера.
- Включение Wi-Fi точки доступа не должно ломать базовую работу row counter в основном меню.
- Сохранение rows после каждого изменения должно быть надежным, но реализация должна учитывать ресурс NVS flash.
- Страница должна быть читаемой на мобильном телефоне.
- Реализация должна использовать ESP-IDF, без Arduino и PlatformIO.

## Вне scope

- Удаленное подключение через домашний Wi-Fi роутер в STA-режиме.
- Пользовательская авторизация, пароль администратора или аккаунты.
- Обновление прошивки через веб-страницу.
- Настройка всех будущих звуков устройства; в scope входит только boot beep.
- Изменение hardware-пинов или добавление нового звукового железа без отдельного подтверждения.
- Синхронизация истории с внешними сервисами.

## Зависимости и ограничения

- Проект использует ESP-IDF для `esp32s3`.
- Устройство: `Waveshare ESP32-S3-LCD-1.69`.
- Кнопка `+` подключена к `GPIO2`, активный уровень `0`.
- Существующее состояние, история и часть данных уже используют NVS.
- Captive portal требует DNS-перехвата и HTTP-сервера на устройстве.
- Режим Wi-Fi AP увеличит энергопотребление и может конфликтовать с режимом сна.

## Открытые вопросы

- Есть ли уже подтвержденный пьезоизлучатель или другой звуковой компонент, и к какому GPIO он подключен?
- Какими должны быть SSID и пароль точки доступа?
- Должен ли AP автоматически выключаться при переходе устройства в сон, если sleep mode включен?
- Какой диапазон и значение по умолчанию нужны для времени до затухания экрана?
- Должно ли касание любой кнопки только пробуждать экран или также выполнять действие кнопки после пробуждения?
- Нужно ли ограничивать частоту NVS-записей для rows, если пользователь быстро нажимает кнопку `+` или `-` много раз подряд?

## Риски и допущения

- Допущение: веб-страница будет обслуживаться с самого устройства в AP-режиме, без внешнего интернета.
- Допущение: ручной адрес страницы настроек будет `192.168.4.1`, если не появится причина изменить стандартный адрес AP.
- Риск: captive portal на разных ОС работает по-разному, поэтому автоматическое открытие страницы не может быть гарантировано на 100%.
- Риск: сохранение rows после каждого изменения увеличивает количество записей во flash; возможно потребуется debounce, coalescing или wear-conscious формат хранения.
- Риск: включенный AP повышает энергопотребление, что может ухудшить работу от батареи.
- Риск: если AP включен, а экран затухает по sleep timeout, пользователь может не увидеть IP-адрес для ручного открытия страницы.
- Риск: если hardware для звука не подтвержден, boot beep можно будет реализовать только после уточнения компонента и GPIO.
